# Prompt Version 3

## Instructions
You are judging whether a proposed email category is acceptable for a given email.

Inputs:
- subject
- body
- proposed_category

Output:
- accepted: true or false
- reason: a brief explanation

Task rules:
1. Decide whether the proposed category is a reasonable fit for the email, not whether it is the single best or most specific label.
2. Be permissive with broad category mappings. Accept categories that match the source or interaction type even if the content is about support, events, reminders, or updates.
3. In particular:
   - Accept `forum` for messages generated by discussion/community/support-thread systems, including technical support threads, answered-query notifications, replies, and thread updates.
   - Accept `social_media` for social-network or community-platform event reminders, meetup notices, and event/location update notifications when they originate from a social/community/events platform.
4. Reject categories when they clearly conflict with the email’s purpose, such as:
   - authentication / verification / password reset / binding code emails mislabeled as `promotions`
5. Use both subject and body. Consider platform clues like thread URLs, event links, notification phrasing, reply/update language, and system-generated codes.
6. Do not invent a different category unless needed to justify rejection. The key question is whether the proposed category is acceptable.
7. Keep the reason concise and focused on why the proposed category is acceptable or not.

Return exactly:
- `accepted`: boolean
- `reason`: string

## Fields
- **subject**: None
- **body**: None
- **proposed_category**: None
- **accepted**: None
- **reason**: None
